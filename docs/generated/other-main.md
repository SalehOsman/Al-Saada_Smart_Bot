# Other — main

The `Other — main` module refers to the `specs/main/plan.md` file, which serves as a **template for implementation plans** within the project's specification workflow. It is not an executable code module but a critical documentation artifact that guides the technical design and development of new features.

## 1. Purpose and Role

The `plan.md` template is the blueprint for detailing the technical approach to a new feature. It is automatically generated by the `/speckit.plan` command and is intended to be filled out by developers during the initial phases of feature development. Its primary goals are:

*   **Standardization**: Ensure a consistent structure for all feature implementation plans.
*   **Clarity**: Provide a clear, comprehensive overview of the technical context, proposed architecture, and development strategy.
*   **Guidance**: Direct developers through the necessary considerations (dependencies, testing, platform, performance, etc.) before writing code.
*   **Gatekeeping**: Include a "Constitution Check" to ensure new features align with project standards and justify any deviations.

This document is central to the `speckit` workflow, bridging the initial feature specification (`spec.md`) with the actual code implementation.

## 2. Key Sections and Their Significance

The `plan.md` template is structured to guide a developer through the process of defining a feature's technical implementation.

### 2.1. Summary

A concise extract from the feature specification, outlining the primary requirement and the high-level technical approach. This section provides immediate context for anyone reviewing the plan.

### 2.2. Technical Context

This is a crucial section for developers to define the foundational technical details of the feature. It covers:

*   **Language/Version**: Specifies the programming language and version to be used.
*   **Primary Dependencies**: Lists key libraries, frameworks, or external services.
*   **Storage**: Details any data persistence mechanisms (e.g., PostgreSQL, files).
*   **Testing**: Outlines the testing framework and strategy (e.g., `pytest`, `XCTest`).
*   **Target Platform**: Defines the deployment environment (e.g., Linux server, iOS).
*   **Project Type**: Categorizes the feature (e.g., library, CLI, web-service).
*   **Performance Goals**: Quantifies expected performance metrics.
*   **Constraints**: Lists specific limitations or requirements (e.g., memory limits, offline capability).
*   **Scale/Scope**: Describes the expected usage or size of the feature.

**Developer Action**: This section requires careful consideration and research (Phase 0 output) to ensure all technical aspects are clearly defined.

### 2.3. Constitution Check

This section acts as a gate, ensuring that the proposed feature adheres to established project "constitution" rules. It must pass before Phase 0 research and be re-checked after Phase 1 design. Any violations must be justified in the "Complexity Tracking" section.

### 2.4. Project Structure

This section is vital for understanding how the new feature will integrate into the existing codebase and how its own internal structure will be organized. It distinguishes between:

*   **Documentation (this feature)**: Details the expected layout of specification-related files for the feature (e.g., `plan.md`, `research.md`, `data-model.md`, `tasks.md`).
*   **Source Code (repository root)**: Provides placeholder options for common project structures (single project, web application, mobile + API). Developers are expected to replace these placeholders with the concrete, real directory layout for the feature, documenting the chosen structure.

**Developer Action**: This section requires a clear decision on the feature's architectural placement and internal organization, including real paths.

### 2.5. Complexity Tracking

This optional section is used *only if* the "Constitution Check" reveals violations that require justification. It provides a structured way to explain why a particular complexity or deviation is necessary and why simpler alternatives were rejected.

## 3. Workflow Integration

The `plan.md` template is an integral part of the `speckit` feature development lifecycle:

```mermaid
graph TD
    A[Feature Idea/Requirement] --> B{speckit.plan command};
    B --> C[specs/[###-feature]/plan.md (Template)];
    C --> D[Developer Fills in Plan];
    D --> E[Phase 0: Research (updates plan)];
    E --> F[Phase 1: Design (updates plan, creates data-model.md, contracts/)];
    F --> G[Phase 2: Implementation (guided by plan, creates tasks.md)];
    G --> H[Code Development];
```

1.  **Generation**: The `/speckit.plan` command initializes the `plan.md` file for a new feature.
2.  **Phase 0 (Research)**: Developers fill in the initial technical context and research findings, updating the `plan.md` and potentially creating `research.md`.
3.  **Phase 1 (Design)**: The plan is refined with detailed design decisions, including data models (`data-model.md`) and API contracts (`contracts/`). The "Constitution Check" is re-evaluated.
4.  **Phase 2 (Implementation)**: The completed `plan.md` serves as the authoritative guide for implementation, leading to the creation of `tasks.md` and the actual source code.

## 4. Contribution Guidelines

When contributing to a new feature, developers should:

*   **Start with `/speckit.plan`**: Always generate the `plan.md` using the specified command.
*   **Be Thorough**: Fill in all sections of the `plan.md` template with accurate and detailed information. Avoid "NEEDS CLARIFICATION" in the final plan.
*   **Document Structure**: Clearly define the chosen project structure for the feature, replacing placeholders with actual paths.
*   **Justify Complexity**: If the feature introduces complexity or deviates from established norms, use the "Complexity Tracking" section to provide clear justifications.
*   **Iterate**: The `plan.md` is a living document during the initial phases. Update it as research and design decisions evolve.

## 5. Call Graph & Execution Flows

As `specs/main/plan.md` is a documentation template and not executable code, there are no associated internal calls, outgoing calls, incoming calls, or execution flows. Its "execution" is conceptual, guiding human developers through a planning process.

# Module Kit V2: Schema-Driven App Factory

## 1. الرؤية والهدف (Vision & Goal)
تحويل `Module Kit` من مجرد أداة لإنشاء قوالب فارغة (Scaffolding Boilerplates) إلى **محرك بناء نماذج متكامل (App Factory)** يعتمد على توليد الكود وقواعد البيانات تلقائياً بناءً على إعدادات المطور، مما يقلل كتابة الكود اليدوي إلى الصفر تقريباً في المهام المعتادة.

## 2. الإجابة على الاستفسارات الفنية (Technical Q&A)

### س1: هل ستقوم المنهجية بإنشاء جدول البيانات وحقوله بشكل صحيح لكل موديول؟
**نعم، بالتأكيد!** 
عند إنشاء موديول جديد وإضافة حقوله عبر الـ CLI، سيقوم الـ Generator بكتابة ملف `schema.prisma` مخصص لهذا الموديول تلقائياً، محوّلاً أنواع البيانات المدخلة في الـ CLI إلى أنواع بيانات Prisma صحيحة (مثال: `Money` يتحول إلى `Float`، `ShortText` إلى `String`، `Photo` إلى `String` لحفظ معرف الصورة). 

### س2: هل يوجد إمكانية للتحكم في جدول البيانات المُنشأ وربطه بعلاقات (Relations) بجداول أخرى؟ (مثال: جلب أسماء العاملين من موديول آخر).
**هذا هو الجزء الأقوى في التحديث القادم!**
نعم، يمكننا بناء ميزة **"البيانات المرجعية" (Relational Fields / Reference Fields)**.
عندما يطلب منك الـ CLI تحديد نوع الحقل، سيكون من ضمن الخيارات: `Reference` أو `Relation`.
- سيطلب منك الـ CLI: **"ما هو الجدول/الموديول الذي تريد الربط به؟"** (ستختار مثلاً: `Employees`).
- سيطلب منك: **"ما الحقل الذي تريد عرضه للمستخدم ليختاره؟"** (ستختار مثلاً: `fullName`).
- **كيف سيتم التنفيذ؟** 
  1. في قاعدة البيانات (`Prisma`): سيصنع السكريبت علاقة حقيقية (Foreign Key) مثل `@relation(fields: [employeeId], references: [id])`.
  2. في البوت (Telegram): عندما يصل المستخدم لهذا السؤال، النظام سيبحث تلقائياً في قاعدة بيانات `Employees`، ويجلب الأسماء، ويعرضها للمستخدم على شكل "أزرار" (Inline Keyboard) ليختار منها اسم العامل مباشرة!

## 3. آلية إنشاء الموديولات المقترحة (Proposed Workflow)

لتحقيق أقصى قدر من المرونة والسرعة، سيتم إنشاء الموديول عبر **ملف إعدادات (Blueprint/Config File)** بدلاً الاعتماد الكلي على حوار طويل جداً في سطر الأوامر (الذي قد يصعب تصحيح الأخطاء فيه أثناء الإدخال).

### الخطوة الأولى: إنشاء ملف المخطط (Blueprint)
يقوم المطور بإنشاء ملف `YAML` أو إعداد سريع عبر الـ CLI، يبدو هكذا:

```yaml
module:
  slug: employee-withdrawal
  name: "سحب سلفة عامل"
  icon: "💰"

fields:
  - name: employeeId
    type: relation
    targetModel: Employee
    displayField: fullName
    label: "من فضلك، اختر اسم العامل:"
    required: true
    
  - name: amount
    type: money
    label: "اكتب المبلغ المراد سحبه:"
    required: true
    
  - name: signatureImage
    type: photo
    label: "ارفع صورة إيصال الاستلام (اختياري)"
    required: false
```

### الخطوة الثانية: تشغيل محرك التوليد (The Generator)
يقوم المطور بكتابة الأمر:
```bash
npm run module:generate employee-withdrawal.yaml
```

**ماذا سيفعل المحرك؟**
1. **Database:** يولد كود الـ `Prisma` كاملاً (مع مفتاح العلاقة Foreign Key).
2. **Logic:** يبني ملف التحقق (Zod Schema) مع السماح بالـ `amount` كنوع رقمي فقط.
3. **UI Engine:** بمجرد دخول المستخدم للموديول، سيتولى "محرك المحادثة" عرض الأسئلة بالترتيب: يجلب قائمة العمال من الداتابيز لتظهر كأزرار، ثم يطلب الرقم المالي، ثم يطلب الصورة مع إضافة زر "تخطي" لأنها `required: false`، وعند التأكيد يحفظها في قاعدة البيانات تلقائياً.

## 4. الفوائد (Benefits of V2 Method)
1. **سرعة الخيال:** الموديول الذي كان يستغرق تصميمه برمجياً ساعات، سيستغرق دقائق لإنشاء ملف إعدادات وتشغيل توليده.
2. **علاقات جداول قوية (Cross-Module Relations):** بفضل خيار `relation`، لن تعيد إدخال بيانات موجودة أصلاً.
3. **صيانة سهلة:** للتعديل، غير حقل `required: true` إلى `false` في ملف الـ Blueprint وأعد التشغيل.

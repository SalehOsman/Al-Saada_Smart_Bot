## نظرة عامة

يعتمد هذا المشروع على منهجية تطوير ثنائية الأدوار، حيث تعمل أداتان من الذكاء الاصطناعي بأدوار متكاملة ومختلفة تماماً. هذا الفصل بين الأدوار يضمن جودة عالية ويمنع الأخطاء من التراكم.

## المراجع والمنهجيات الإلزامية (Mandatory Methodologies)

يُحظر تماماً على المستشار التقني أو المنفذ الآلي (AI) الانحراف عن مسارات العمل القياسية أدناه. أي تصرف ارتجالي غير مدعوم بهذه المراجع يُعتبر خرقاً للمنهجية:

1. **إطار عمل SpecKit (`github/spec-kit`)**:
   - المنفذ (`/speckit.implement`) لا يجوز له تعديل ملفات التوثيق (`spec.md`، `tasks.md`، إلخ) بأوامر جانبية معقدة.
   - البوابة التحليلية (`speckit.analyze`) هي ممر إلزامي قبل أي تنفيذ.
   
2. **مهارات Antigravity (`sickn33/antigravity-awesome-skills`)**:
   - يجب على المستشار التقني استخدام المهارات الهندسية والاستشارية المناسبة (مثل `architect`, `plan-writing`, `conductor`) لتصميم المهام واضحة الحدود.
   - يجب عدم اللجوء لأوامر `bash` معقدة أو للـ micro-management لمعالجة ملفات التوثيق إن لم تكن هناك ضرورة قصوى خارج نطاق قدرات الـ AI الطبيعية.

## الأدوار

### الدور الأول: المنفّذ (Executor)

**الوظيفة:** كتابة الكود وإنشاء الملفات وتنفيذ الأوامر مباشرة على المشروع.

**المسؤوليات:**
- إنشاء الملفات والمجلدات
- كتابة الكود البرمجي
- تشغيل أوامر الـ Terminal (npm, git, docker, prisma, etc.)
- تنفيذ الـ commits
- تشغيل الاختبارات
- تشغيل أوامر Spec Kit (constitution, specify, plan, tasks, implement, analyze)

**القيود:**
- لا يُنفّذ إلا ما يُطلب منه بشكل صريح
- لا يتجاوز نطاق المهمة المحددة
- ينتظر الموافقة قبل تنفيذ أي إجراء لا رجعة فيه (مثل push, delete)

**صيغة الأوامر:**
```
/speckit.implement Implement ONLY tasks [IDs]. [تفاصيل دقيقة لما يجب إنشاؤه]. Commit with: "[رسالة]"
```

---

### الدور الثاني: المستشار التقني والمراجع (Technical Advisor & Reviewer)

**الوظيفة:** مراجعة كل مخرجات المنفّذ، اكتشاف الأخطاء، واقتراح التحسينات قبل المتابعة.

**المسؤوليات:**

#### 1. التخطيط والتوجيه
- تجهيز ملفات المدخلات (input files) للمنفّذ
- شرح النطاق الكامل للمرحلة القادمة (Phase Scope) بوضوح لصاحب المشروع قبل بدء التنفيذ.
- صياغة الأوامر الدقيقة التي يجب إرسالها للمنفّذ
- تحديد ترتيب المهام والتبعيات بينها
- منع المنفّذ من تنفيذ خطوات خاطئة أو سابقة لأوانها

#### 2. المراجعة الفورية
- قراءة كل ملف يُنشئه المنفّذ بعد كل خطوة
- مقارنة المخرجات مع المواصفات والدستور
- اكتشاف الأخطاء التقنية (مثل: nanoid vs cuid في Prisma)
- اكتشاف عدم التناسق بين الملفات (مثل: port مختلف بين docker-compose و .env)
- التأكد من أن التعليمات الموجهة لا تتعارض مع القرارات المعمارية

#### 3. الإصلاح المباشر
- الإبلاغ عن أي خطأ فوراً — لا تأجيل أبداً
- **يُحظر** تعديل أي ملف مباشرة إلا بطلب صريح من صاحب المشروع
- تحديث ملفات التوثيق (spec, plan, data-model) لتعكس أي تغيير — بطلب صريح فقط
- التأكد من التناسق الشامل عبر كل ملفات المشروع بعد كل إصلاح

#### 4. التحقق من التناسق
- بعد كل إصلاح: فحص شامل لكل الملفات المتأثرة
- التأكد من عدم وجود آثار للنص القديم في أي ملف
- التحقق من توافق الكود مع التوثيق والعكس

#### 5. اتخاذ القرارات المعمارية
- تقييم المقترحات التقنية (مثل: Hybrid vs Zero-Code)
- اقتراح تعديلات على الدستور عند الحاجة
- الموازنة بين المثالية والواقعية (Time to Market)

## القواعد الذهبية

### 1. مبدأ الخطوتين (Two-Task Rule)
- لا تُنفّذ أكثر من مهمتين في كل دورة
- بعد كل دورة: مراجعة كاملة قبل المتابعة
- هذا يمنع تراكم الأخطاء ويسهّل اكتشافها

### 2. مبدأ الإصلاح الفوري (Fix-Now Principle)
- أي خطأ يُكتشف يُصلح فوراً في نفس اللحظة
- لا يوجد "نصلحه لاحقاً" — هذا ممنوع
- الإصلاح يشمل: الكود + التوثيق + أي ملف متأثر

### 3. مبدأ التناسق الشامل (Full Consistency Principle)
- بعد أي تعديل: فحص كل الملفات المتأثرة
- يجب أن تتطابق المعلومات عبر: الدستور، المواصفات، الخطة، نموذج البيانات، المهام، والكود
- لا يُقبل أي تعارض بين ملفين

### 4. مبدأ الأمر الصريح (Explicit Command Principle)
- المنفّذ لا يجتهد — ينفّذ فقط ما يُطلب
- الأوامر يجب أن تكون دقيقة ومحددة
- تشمل: ما يُنشأ، ما لا يُنشأ، رسالة الـ commit

### 5. مبدأ المنع الاستباقي (Preemptive Prevention)
- المستشار يمنع المنفّذ من:
  - تنفيذ `push` بدون remote
  - حذف commits صحيحة
  - تنفيذ خطوات خارج الترتيب
  - إعادة توليد ملفات كاملة بدلاً من تعديلات مستهدفة

### 7. مبدأ i18n-Only — لا نصوص مُضمَّنة في الكود (i18n-Only Text Principle)

يُحظر تماماً كتابة أي نص يظهر للمستخدم مباشرةً داخل ملفات TypeScript أو أي ملف كود آخر في المشروع. ينطبق هذا الحظر على:
- الرسائل والتعليمات (عربية أو إنجليزية)
- نصوص الأزرار والقوائم
- رسائل الخطأ والتحقق
- قيم العرض (مثل: "ذكر", "أنثى", "غير محدد", "قيد المراجعة")
- أي نص يصل إلى المستخدم عبر الواجهة

**القاعدة التطبيقية:**
- كل نص للمستخدم يُعرَّف حصراً في ملفي `ar.ftl` و `en.ftl`
- الكود يستخدم مفاتيح الترجمة فقط: `ctx.t('key_name')`
- الدوال التي تُصنّف البيانات (مثل `formatGender`) تُرجع مفتاح i18n وليس نصاً مباشراً
- عند الإضافة أو التعديل: يُضاف المفتاح في كلا الملفين قبل استخدامه في الكود

**المخالفة:** أي `ctx.reply('نص مباشر')` أو `return 'ذكر'` أو ما شابه تُعتبر مخالفة صريحة لهذا المبدأ وتستوجب الإصلاح الفوري.

### 10. مبدأ الإكمال الكامل قبل التقدم (Zero-Defect Gate Principle)

**القرار المتخذ بتاريخ 2026-02-24:**
لا يجوز الانتقال إلى أي مرحلة أو خطوة جديدة إلا بعد حل جميع مشاكل المرحلة الحالية بالكامل.

**القاعدة:**
> لا تقدّم بكود معطوب — كل خطوة يجب أن تكون صحيحة 100% قبل الانتقال للخطوة التالية.

**ينطبق على:**
- جميع مخرجات `/speckit.analyze` — يجب حل كل المشاكل قبل `/speckit.implement`
- جميع الاختبارات — يجب أن تكون 100% passing قبل الانتقال لـ Phase التالية
- جميع مشاكل TypeScript / Linting — يجب حلها قبل الـ commit
- جميع التعارضات بين الملفات (spec, plan, tasks, constitution) — يجب توحيدها أولاً
- جميع مشاكل `/speckit.analyze` — CRITICAL أولاً، ثم HIGH، ثم MEDIUM

**الترتيب الإلزامي لكل Phase:**
```
1. /speckit.analyze → لا صفر مشاكل = لا تنفيذ
2. إصلاح CRITICAL → إصلاح HIGH → إصلاح MEDIUM
3. /speckit.analyze مرة ثانية → تأكيد صفر مشاكل
4. /speckit.implement → التنفيذ
5. اختبارات 100% passing
6. /speckit.analyze نهائي → تأكيد
7. انتقال للـ Phase التالية
```

**المخالفة:** الانتقال لخطوة جديدة بوجود أي مشكلة غير محلولة يُعتبر خرقاً صريحاً يستوجب العودة وإصلاح كل شيء قبل المتابعة.

### 8. مبدأ الوحدات الجاهزة أولاً (Shared-First Principle)

قبل كتابة أي تدفق أو خطوة داخل تدفق، يجب السؤال: **"هل هذا الكود يمكن أن يُستخدم في مكان آخر؟"**

إذا كانت الإجابة نعم — يُنشأ كوحدة جاهزة في `bot/utils/` أولاً، ثم يُستدعى من التدفق. يُحظر تكرار نفس المنطق في ملفين مختلفين.

**ينطبق هذا على:**
- أي خطوة لجمع مدخل من المستخدم (اسم، هاتف، تاريخ، رقم، إلخ)
- أي دالة تنسيق أو تحويل بيانات (formatDate, formatRole, formatStatus)
- أي منطق تحقق يتكرر في أكثر من تدفق
- أي رسالة تأكيد أو إلغاء أو خطأ مشتركة بين التدفقات
- أي إشعار يُرسل لأكثر من تدفق (notifyAdmins, notifyUser)

**الوحدات الجاهزة الحالية في `packages/core/src/bot/utils/`:**
- `conversation.ts` — تتبع الرسائل، الانتظار، التأكيد، الإلغاء
- `user-inputs.ts` — جمع الاسم، الهاتف، الرقم القومي، توليد اسم الشهرة
- `formatters.ts` — تنسيق التواريخ، مفاتيح i18n للجنس والحالة، إشعارات الأدمنية

**القاعدة التطبيقية:**
1. قبل كتابة أي تدفق جديد — راجع `bot/utils/` أولاً
2. استخدم الموجود مباشرة بدلاً من إعادة الكتابة
3. إذا احتجت منطقاً جديداً مشتركاً — أضفه لـ `bot/utils/` أولاً ثم استدعه
4. لا يُقبل وجود نفس المنطق في ملفين — هذا تكرار ممنوع

**المخالفة:** أي كود متكرر بين تدفقين يُعتبر مخالفة صريحة ويستوجب الإصلاح الفوري بالاستخراج إلى `bot/utils/`.

### 9. مبدأ البنية الجاهزة للذكاء الاصطناعي (AI-Ready Architecture Principle)

**القرار المتخذ بتاريخ 2026-02-24:**
المشروع يُبنى من البداية ليستقبل AI في أقرب فرصة ممكنة — وليس كإضافة لاحقة.

**التقنيات المختارة:**
- **LLM:** Qwen2.5:7b عبر Ollama (محلي — بيانات الشركة لا تغادر الخادم)
- **Embeddings:** nomic-embed-text (محلي، دعم عربي)
- **Vector DB:** pgvector فوق PostgreSQL الموجود (لا DB إضافي)
- **Package:** packages/ai-assistant/ (منفصل عن packages/core/)

**الجدول الزمني الموازي:**

| المرحلة | ما يُنفَّذ | يوازي |
|---|---|---|
| A | pgvector + Ollama في docker-compose، هيكل packages/ai-assistant/ فارغ | Phase 6-7 |
| B | جدول Embedding في Prisma، Embedding Service، RAG Service، LLM Client | Phase 8-9 |
| C | تكامل كامل مع البوت، RBAC على RAG، UI المحادثة | بعد Phase 11 |

**قواعد AI-RBAC:**
- SUPER_ADMIN → يرى كل البيانات
- ADMIN → يرى بيانات قسمه فقط (AdminScope)
- EMPLOYEE → يرى بياناته الشخصية فقط
- VISITOR → مرفوض قبل الوصول للـ RAG

**مصدر التفاصيل الكاملة:** `specs/002-ai-assistant/spec.md` + `plan.md` + `tasks.md`

**المخالفة:** بناء AI بدون RBAC filter أو إرسال بيانات الشركة لـ APIs خارجية يُعتبر خرقاً أمنياً ويستوجب الإصلاح الفوري.

### 6. مبدأ البطاقة التوضيحية (Command Brief Principle)
- قبل إرسال أي أمر للمنفّذ، يُلزم المستشار بكتابة **بطاقة توضيحية** لصاحب المشروع
- الهدف: تمكين صاحب المشروع من فهم ما سيحدث ومناقشته قبل التنفيذ
- لا يُرسل أي أمر للمنفّذ دون موافقة صاحب المشروع على البطاقة

**صيغة البطاقة التوضيحية الإلزامية:**
```
📋 بطاقة الأمر — [رقم المهمة / اسمها]

🎯 الهدف:
[ماذا سيحقق هذا الأمر بكلمات بسيطة]

📁 الملفات المتأثرة:
- [مسار الملف] — [سيُنشأ / سيُعدَّل / سيُحذف]

⚙️ ما سيفعله المنفّذ تحديداً:
1. [خطوة]
2. [خطوة]

⚠️ ملاحظات / مخاطر:
[أي تحذيرات أو قرارات تحتاج موافقة — أو "لا توجد"]

✅ للموافقة: أرسل الأمر التالي للمنفّذ
```

## دورة المهمة الواحدة

```
1. المستشار يكتب البطاقة التوضيحية ──▶ صاحب المشروع يراجع ويوافق
2. المستشار يصيغ الأمر الدقيق ──▶ صاحب المشروع ينسخه للمنفّذ
3. المنفّذ ينفّذ ويعرض النتيجة
4. صاحب المشروع ينقل النتيجة للمستشار
5. المستشار يقرأ الملفات الفعلية (وليس ملخص المنفّذ)
6. إذا وُجدت مشاكل ──▶ المستشار يصلحها مباشرة
7. إذا سليم ──▶ المستشار يصيغ البطاقة التوضيحية للأمر التالي
```

**ملاحظة مهمة جداً (قاعدة المراجعة الحقيقية):** يجب على المستشار التقني قراءة ومراجعة التعديلات من واقع الملفات الفعلية في النظام (عبر قراءة الملفات مباشرة) ويُحظر عليه الاعتماد أو الموافقة بناءً على رسالة المخرجات (Output message / Summary) المرسلة من المنفذ فقط، وذلك لضمان اكتشاف أي أخطاء تراكمية أو مخفية في الكود.

## متى يتدخل المستشار مباشرة؟

| الحالة | التصرف |
|--------|--------|
| خطأ تقني في الكود | يُبلّغ صاحب المشروع ويقترح الإصلاح — لا تعديل بدون طلب صريح |
| عدم تناسق بين الملفات | يُبلّغ صاحب المشروع ويقترح التعديلات المطلوبة |
| المنفّذ يقترح خطوة خاطئة | يمنعه ويوضح السبب |
| تعديل معماري مطلوب | يجهّز ملف التعديل ويوجّه المنفّذ |
| المنفّذ يتجاوز النطاق | يوقفه فوراً |
| سؤال يحتاج قرار | يعرض الخيارات على صاحب المشروع |

## مستويات التحقق

### بعد كل مهمة (Task Level)
- قراءة الملف المُنشأ
- التحقق من الـ imports والـ exports
- التحقق من التوافق مع المواصفات

### بعد كل مرحلة (Phase Level)
- فحص التناسق الشامل (Consistency Check)
- مقارنة الكود مع التوثيق
- التأكد من أن كل الملفات المتأثرة محدّثة

### بعد كل إصلاح (Fix Level)
- بحث شامل عن النمط القديم في كل المشروع
- التأكد من عدم وجود آثار متبقية
- تحديث التوثيق ليعكس الإصلاح

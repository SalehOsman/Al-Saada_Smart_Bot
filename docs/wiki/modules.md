# نظام الوحدات (Modules System)

الطبقة الثالثة في بنية مشروع Al-Saada Smart Bot هي الـ **Modules**، وهي تعتمد بشكل تام على مفهوم **Config-First Architecture** (البناء اعتماداً على الإعدادات).

## ما هي الوحدة (Module)؟
الوحدة هي تطبيق مصغر، مثل "إدارة الموظفين" أو "تسجيل عهدة". بدلاً من كتابة كود برمجي جديد لإنشاء شاشات ومحادثات لكل جزء، نقوم بإنشاء ملفات إعدادات (Configuration) يقوم "محرك التدفق" (Flow Engine) بقراءتها وتشغيلها داخل Telegram.

## هيكل الوحدة القياسي
أي وحدة جديدة تتكون من الملفات التالية:
- `module.config.ts`: التعريف الأساسي (الاسم، القسم، الأيقونة).
- `add.flow.ts`: مسار الإضافة (خطوات إدخال البيانات).
- `edit.flow.ts`: مسار التعديل.
- `list.config.ts`: إعدادات عرض القوائم والبحث.
- `view.config.ts`: إعدادات عرض السجل الواحد.
- `schema.prisma`: تعريف جدول قاعدة البيانات الخاص بالوحدة (يتم تجميعه في الـ Schema العام).

## ما هو الـ Flow Block؟
لبناء مسار إدخال (مثلاً `add.flow.ts`)، نقوم بربط مجموعة من "الكتل" (Blocks).
محرك التدفق يحتوي على كتل جاهزة، مثل:
- `text_input`: لإدخال نص.
- `national_id`: لطلب الرقم القومي المصري والتحقق منه واستخراج تاريخ الميلاد.
- `phone_input`: لطلب رقم هاتف مصري.

### مثال على إعدادات مسار إضافة (Add Flow):
```ts
steps: [
  { block: 'text_input', field: 'fullName' },
  { block: 'national_id', field: 'nationalId' },
  { block: 'phone_input', field: 'phoneNumber' },
  { block: 'confirm', showSummary: true },
]
```

## الـ Hooks (حالات الاستثناء)
إذا كانت الوحدة تتطلب عملية معقدة لا يمكن التعبير عنها في ملفات الإعدادات (مثل طلب بيانات من API خارجي لحساب تكلفة، أو عملية حسابية معقدة قبل الحفظ)، يتم اللجوء للـ **Hooks**.
- `beforeValidate`, `beforeSave`, `afterSave`, `beforeDelete`, `onApproval`

**القاعدة الذهبية (90/10 Rule):** يجب أن يغطي الـ Config حوالي 90% من عمل الوحدة المعنية، ويُستخدم الـ Hook لـ 10% كحد أقصى.

# نظرة عامة (Overview)

## 1. هوية المشروع
- **الاسم:** Al-Saada Smart Bot (بوت السعادة الذكي)
- **الهدف:** منصة ذكية لإدارة الأعمال والمؤسسات المصرية عبر Telegram.
- **الفلسفة الأساسية:** المنصة عبارة عن محرك فارغ وذكي للروتينيات (Flow Engine) يقوم بتحميل وحدات (Modules) مبنية بالكامل اعتماداً على الإعدادات (Config-Driven) بدلاً من البرمجة التقليدية المعقدة (Hardcoding).
- **الذكاء الاصطناعي:** يتضمن المشروع مساعداً تشغيلياً ذكياً (AI Operational Assistant) في المرحلة الرابعة يعتمد على نماذج محلية (مثل Qwen3-8B) مع تقنية RAG للتفاعل مع بيانات الشركة باللغة العربية نصاً وصوتاً.

---

## 2. البنية المعمارية (Three-Layer Architecture)

يعتمد إطار العمل على 3 طبقات رئيسية:

### الطبقة الأولى: Platform Core (المنصة الأساسية)
وهي الأساس الثابت الذي لا يتغير باختلاف نوع البزنس. تحتوي على:
- الاتصال مع قواعد البيانات (PostgreSQL + Prisma) وذاكرة التخزين المؤقت (Redis).
- إعداد بوت Telegram باستخدام إطار عمل (grammY).
- نظام الصلاحيات (RBAC) بـ 4 أدوار: `SUPER_ADMIN`, `ADMIN`, `EMPLOYEE`, `VISITOR`.
- نظام اكتشاف وتحميل الوحدات الديناميكي (Module Loader).
- إدارة الأقسام (Sections) وسجل الحركات (Audit Logging) ونظام التنبيهات.
- **أدوات المحادثة المشتركة** (`bot/utils/`) — وحدات مُعاد استخدامها عبر جميع تدفقات البوت.

### الطبقة الثانية: Flow Engine (محرك التدفق)
القلب النابض للمنصة، يحول ملفات الإعدادات إلى شاشات Telegram تعمل بكفاءة. يحتوي على:
- **الـ Flow Blocks:** وحدات بناء جاهزة لإدخال البيانات (مثل: نص، رقم، تاريخ، رقم هاتف مصري، رقم قومي، مواقع).
- محرك القوائم ومحرك التقارير ومحرك البحث.

### الطبقة الثالثة: Modules (الوحدات)
هي الميزات الفعلية للشركة (مثل: الموارد البشرية، المصروفات، حركة السيارات).
- **مبدأ 90/10:** تُبنى الوحدة بنسبة 90% كملفات إعدادات (Config)، و10% كأكواد برمجية (Hooks) في الحالات المعقدة (مثل حسابات الضرائب).

## 3. حالة المشروع الحالية (Project Status)

**المرحلة الحالية:** Phase 1 — Platform Core (قيد التطوير)

### ما تم إنجازه ✅

**البنية التحتية:**
- هيكلة Monorepo (`packages/core`, `packages/validators`)
- Docker Compose مع PostgreSQL 16 و Redis 7
- Prisma ORM متصل بقاعدة البيانات
- Redis متصل لإدارة الجلسات
- Pino Logger مع دعم عربي
- Health Check endpoints
- Graceful shutdown

**نظام المستخدمين والتسجيل:**
- تدفق `/start` الموحد — Bootstrap + Join Request في محادثة واحدة
- Bootstrap Lock: أول مستخدم يطابق `INITIAL_SUPER_ADMIN_ID` يصبح SUPER_ADMIN تلقائياً
- تدفق طلب الانضمام بـ 4 خطوات: الاسم، اسم الشهرة، الهاتف، الرقم القومي
- دعم الأرقام العربية (٠-٩) في جميع حقول الأرقام
- استخراج تاريخ الميلاد والجنس من الرقم القومي تلقائياً
- توليد اسم الشهرة تلقائياً من أول كلمتين مع دعم الأسماء المركبة (عبد، أبو، آل)
- القوائم حسب الدور (SUPER_ADMIN / ADMIN / EMPLOYEE / VISITOR)
- Audit logging لعمليات Bootstrap

**أدوات المحادثة المشتركة (`packages/core/src/bot/utils/`):**
- `conversation.ts` — تتبع الرسائل، أدوات الانتظار، الإلغاء، التأكيد
- `user-inputs.ts` — جامعات المدخلات المصرية مع تطبيع الأرقام العربية
- `formatters.ts` — تنسيق التواريخ، مفاتيح i18n للجنس، إشعارات الأدمنية

**التحقق من المدخلات (`packages/validators/`):**
- `egyptianPhoneNumber()` — رقم الهاتف المصري (010/011/012/015)
- `egyptianNationalId()` — الرقم القومي المصري (14 رقماً)
- `extractEgyptianNationalIdInfo()` — استخراج تاريخ الميلاد والجنس

**الترجمة والنصوص:**
- نظام i18n-Only: **لا نصوص مُضمَّنة في الكود** — كل النصوص في `ar.ftl` و `en.ftl`
- الدوال التي تُصنّف البيانات تُرجع مفاتيح i18n وليس نصوصاً مباشرة
- دعم كامل للعربية والإنجليزية

### ما لم يُنجز بعد ⏳

- نظام الإشعارات عبر BullMQ (T053/T054)
- RBAC كامل مع `canAccess()` و AdminScope
- إدارة الأقسام (Sections) عبر البوت
- نظام الصيانة (Maintenance Mode)
- Module Loader الديناميكي
- اختبارات تكامل لتدفق التسجيل

## 5. المبادئ الأساسية للتطوير

1. **i18n-Only:** كل نص يظهر للمستخدم يجب أن يكون في `.ftl` فقط — `ctx.t('key')` لا `ctx.reply('نص مباشر')`
2. **Config-Driven:** 90% من وظائف الوحدات كإعدادات، 10% كـ Hooks
3. **Platform-First:** Layer 1 + Layer 2 يجب أن تكتمل قبل أي وحدة
4. **Egyptian Context:** جميع المُدققات تدعم الصيغ المصرية
5. **Shared Utils:** أي كود متكرر بين التدفقات ينتقل إلى `bot/utils/`
